The RNN Bottleneck

How recurrent networks tried to handle sequences of any length — and where they broke

Where we left off

Lesson 1 ended with an unsolved problem: neural language models with word embeddings could generalize across meaning, but they still read a fixed-size window of previous words — typically five or ten. That's fine for predicting the next word in a sentence. It completely breaks down for anything longer.

Consider machine translation. To translate a paragraph, you need to hold the whole source sentence in your head before producing the first target word. A fixed window can't do that. You need something that can read a sequence of arbitrary length and remember what it read.

That's what the recurrent neural network (RNN) was designed to do.

What "recurrent" means: a loop through time

An RNN processes a sequence one token at a time, left to right. At each step, it does two things:

  1. Reads the current token's embedding.
  2. Reads a hidden state — a fixed-size vector carried forward from the previous step.

It combines those two inputs, produces a new hidden state, and passes it to the next step. Like this:

"the" "cat" "sat" "on" "the" │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ [h₀] → [RNN] → [h₁] → [RNN] → [h₂] → [RNN] → [h₃] → [RNN] → [h₄] → [RNN] → [h₅] │ predict next word

Each hidden state hₙ is supposed to be a compressed summary of everything the model has read up to that point. After reading "the cat sat on the", the hidden state h₅ should encode — somehow — the full context needed to predict what comes next.

This is genuinely elegant. The same RNN cell, reused at every step, can handle sequences of any length: 5 words, 50 words, 500 words. You just loop it more times.

Sequence-to-sequence: the encoder-decoder idea

In 2014, two teams independently published what became known as the sequence-to-sequence (seq2seq) architecture. The idea: use two RNNs — one to read, one to write.

ENCODER DECODER ─────── ─────── "the" "cat" "sat" "le" "chat" "était" "assis" │ │ │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ ▼ ▼ [RNN] → [RNN] → [RNN] ──► context ──► [RNN] → [RNN] → [RNN] → [RNN] vector (fixed size!)

The encoder reads the full source sentence, token by token, and compresses it into a single fixed-size vector — the context vector. Then the decoder takes that one vector and generates the target sentence, one token at a time.

This worked. It was state of the art for machine translation in 2014–2015. But it had a problem baked into its design.

The bottleneck: compressing everything into one vector

The context vector is a fixed-size vector — say, 512 numbers. That's it. Everything the encoder learned about the source sentence must be compressed into those 512 numbers before the decoder sees a single token.

For short sentences, this is fine. For long ones, it's a disaster. Consider translating a paragraph. The encoder has to compress: the subject, the verb, the object, every adjective, the overall rhetorical structure, any pronouns and their antecedents — all of it — into 512 numbers. Then the decoder has to reconstruct the target language entirely from that single bottleneck.

Analogy: imagine you had to read a whole legal contract, then close the document, and write a translation from memory — but your memory is limited to exactly one index card. Short texts: fine. Long texts: you'll lose details.

This showed up empirically: seq2seq translation quality degraded noticeably as source sentences got longer.

The other problem: vanishing gradients

There's a second, related problem inside the RNN itself.

Training a neural network means adjusting its weights based on errors. The adjustment signal — the gradient — flows backward through the network from the output to every earlier weight. In an RNN, that signal has to travel back through every single time step.

Each step slightly weakens or amplifies the signal (by multiplying by numbers that are usually less than 1). Across 50 or 100 steps, the signal shrinks to near-zero before it reaches the early tokens. The model effectively stops learning long-range dependencies because the training signal can't reach far enough back.

"The contract signed in 1987, which covered all parties, was ___" step 1 step 2 ... step 8 step 9 step 10 "The" "contract" "was" "finally" "void" │ │ │ │ │ │ │ │ │ │ gradient gradient gradient ← ← ← ← gradient (tiny) (tiny) (small) (starts here) By the time the gradient reaches "contract", it's nearly zero. The model can't learn that "was" relates to "contract".

LSTMs (Long Short-Term Memory networks, 1997) and GRUs (Gated Recurrent Units, 2014) were architectural improvements that helped gradients flow further back. They used learned "gates" to decide what to keep and what to forget in the hidden state. They helped — but the fundamental problem remained for very long sequences.

The third problem: RNNs are inherently sequential

RNNs have a structural constraint that matters enormously in practice: step n can't run until step n−1 is complete, because step n needs the hidden state produced by step n−1.

Modern hardware — GPUs, TPUs — is built for parallelism. You want to process thousands of operations simultaneously. But an RNN on a 100-word sentence requires exactly 100 sequential steps, one after another. You can't split that work across GPU cores.

This meant RNNs were slow to train and slow to scale. As researchers wanted to use larger datasets and longer sequences, the sequential constraint became a real wall.

The setup: what we actually need

By 2015, the problems were clear. What would an ideal architecture do?

  • No fixed context window. Every output token should be able to look at every input token directly — not through a bottleneck.
  • No sequential dependency. The architecture should be parallelizable so it can run fast on modern hardware.
  • No vanishing gradients. The training signal should reach every part of the input, regardless of how far back it is.

Attention mechanisms were the first step toward all three. That's Lesson 3.

The arc so far: N-grams → fixed windows (can't generalize). Word embeddings → meaning in vectors (can generalize, still fixed windows). RNNs → arbitrary-length sequences (but bottleneck, vanishing gradients, no parallelism). Next: attention as a direct fix to the bottleneck.
